Premissas técnicas

  • P3 — O repositório deve estar no GitHub (público ou privado — Railway suporta ambos)
  • P4 — Branch principal (master) é a branch de produção. Push nela = novo deploy automático nos três serviços
  • P5 — O banco atual é SQLite local (contatos.db). Os dados de desenvolvimento não serão migrados — o banco Railway PostgreSQL iniciará vazio e as migrations do Alembic criarão as tabelas automaticamente
  • P6 — Será necessário criar o primeiro usuário administrador manualmente após o deploy (o sistema possui autenticação JWT sem cadastro público de admin)
  • P7 — Diferentemente do Render free tier, o Railway não hiberna os serviços. O backend responde instantaneamente a qualquer hora

Arquitetura final

GitHub (branch master) │ ▼ deploy automático Railway Project: "agentic-squad" │ ├── Service: backend FastAPI / uvicorn │ └── URL pública https://backend-xxxx.up.railway.app │ │ rede privada Railway (mais rápido, sem tráfego externo) │ ▼ ├── Service: postgresql PostgreSQL 15 gerenciado │ └── DATABASE_URL injetado automaticamente pelo Railway │ └── Service: frontend Next.js 14 └── URL pública https://frontend-xxxx.up.railway.app │ NEXT_PUBLIC_API_URL → backend URL pública ▼ FastAPI backend

Vantagem chave: backend e banco se comunicam pela rede privada interna do Railway — mais rápido e sem tráfego externo. Na opção Vercel+Render+Supabase esse caminho ocorre pela internet pública.

Etapa 1 — Preparação local do código

Garantir que o código esteja pronto para produção antes de qualquer deploy

~45–60 min Claude Code executa

1.1 — Ajuste do alembic/env.py para usar variável de ambiente

O Alembic atualmente tem a URL SQLite hardcoded no alembic.ini. Será alterado o alembic/env.py para ler DATABASE_URL do ambiente em tempo de execução. Railway injeta DATABASE_URL automaticamente quando o serviço PostgreSQL é vinculado ao backend — sem configuração manual necessária. Sem esse ajuste, as migrations falham ou criam tabelas no arquivo SQLite local em vez do PostgreSQL.

1.2 — Adicionar CORS ao app/main.py

O frontend no Railway (domínio *.up.railway.app) precisa de permissão explícita para chamar a API. Será adicionado o middleware CORSMiddleware do FastAPI, lendo CORS_ORIGINS de variável de ambiente. Sem isso, o browser bloqueia todas as requisições — o app fica inacessível para o usuário final.

1.3 — Criar Procfile no diretório backend/

Railway detecta o Procfile automaticamente para saber como iniciar a aplicação Python.

web: uvicorn app.main:app --host 0.0.0.0 --port $PORT

1.4 — Criar railway.toml no diretório backend/ (recomendado)

Arquivo de configuração declarativa do Railway. Permite definir build command, start command e health check diretamente no repositório. Facilita reproduzir o setup em outro ambiente ou onboarding de novos membros.

1.5 — Criar .env.example em backend/ e frontend/

Documentação de referência com todas as variáveis necessárias (sem valores reais). Servirá como guia ao configurar as variáveis no painel Railway.

1.6 — Verificar .gitignore

Confirmar que .env e contatos.db não estão sendo versionados. Previne exposição acidental de segredos no GitHub.

1.7 — Commit e push das alterações

Criar commit com todas as mudanças de preparação e push para master no GitHub.

Etapa 2 — Configuração do projeto Railway

Criar o projeto e os serviços na plataforma

~15–20 min Você executa

2.1 — Criar conta e assinar plano Hobby

  • Acessar railway.app → "Sign in with GitHub"
  • Após login: Settings → Billing → Upgrade to Hobby ($5/mês)
  • Cadastrar cartão de crédito
  • Nota: o plano Trial ($5 crédito único, sem cartão) pode ser usado para testes, mas os serviços param quando o crédito acaba

2.2 — Criar o projeto Railway

  • Dashboard → "New Project" → "Deploy from GitHub repo"
  • Selecionar o repositório
  • Cancelar o wizard automático e configurar manualmente cada serviço (mais controle)

2.3 — Adicionar serviço PostgreSQL

  • Dentro do projeto → "New Service" → "Database" → "Add PostgreSQL"
  • Railway provisiona o banco em ~30 segundos
  • A DATABASE_URL é gerada automaticamente e ficará disponível para injeção nos outros serviços via referência

Etapa 3 — Deploy do backend (FastAPI)

Colocar a API FastAPI online no Railway

~20–25 min Você executa

3.1 — Criar serviço de backend

No projeto Railway → "New Service" → "GitHub Repo". Configurar:

CampoValor
Service Namebackend
Root Directorybackend
Build Commandpip install -r requirements.txt
Start Commanduvicorn app.main:app --host 0.0.0.0 --port $PORT

3.2 — Vincular o PostgreSQL ao backend

Na aba "Variables" do serviço backend → "Add Reference" → selecionar o serviço PostgreSQL e adicionar a variável DATABASE_URL. Railway injeta automaticamente a connection string correta — sem copiar/colar manual.

3.3 — Configurar variáveis de ambiente do backend

VariávelValorObservação
DATABASE_URL(via referência)Adicionada na etapa 3.2 — não inserir manualmente
SECRET_KEYstring aleatória 64 charsGerar com python -c "import secrets; print(secrets.token_hex(32))"
ENVproductionAtiva logs JSON estruturados
ALGORITHMHS256Algoritmo JWT
ACCESS_TOKEN_EXPIRE_MINUTES60Expiração do token
CORS_ORIGINSpreencher após Etapa 4URL do frontend gerada pelo Railway

3.4 — Gerar URL pública do backend

Aba "Settings" → "Networking" → "Generate Domain". Railway gera: https://backend-xxxx.up.railway.app. Anotar essa URL — será usada como NEXT_PUBLIC_API_URL no frontend.

3.5 — Primeiro deploy e execução das migrations

O deploy inicia automaticamente após a configuração (~3–5 minutos). Após build bem-sucedido, executar as migrations via Railway Shell:

alembic upgrade head

Isso criará todas as tabelas no PostgreSQL. O projeto possui 5 migrations: initial, deletado_em, auditoria, indices, indices sort/filter.

3.6 — Verificar a API

Acessar https://backend-xxxx.up.railway.app/docs → Swagger UI deve aparecer e confirmar que a API está respondendo.

3.7 — Criar o primeiro usuário administrador

Via Railway Shell no serviço backend, executar script de seed ou chamada ao endpoint de criação de usuário. O banco inicia vazio — sem esse passo não é possível fazer login no sistema.

Etapa 6 — Automação de migrations (opcional, recomendada)

Garantir que novas migrations rodem automaticamente a cada deploy

~15–20 min Claude Code executa

6.1 — Adicionar pre-deploy command no railway.toml

Configurar o Railway para rodar alembic upgrade head antes de cada novo deploy. Isso garante que futuras alterações no banco sejam aplicadas automaticamente. Sem isso, cada nova migration precisaria ser rodada manualmente via Shell.

Etapa 7 — Hardening básico (opcional, pós-validação)

Melhorias de segurança e operabilidade — não bloqueante

~30–60 min Opcional

7.1 — Domínio customizado

Railway permite adicionar domínio próprio gratuitamente no plano Hobby. Precisaria adquirir um domínio (~R$40–80/ano). Configuração: painel Railway → Settings → Custom Domain.

7.2 — GitHub Actions para CI

Adicionar workflow que roda pytest (backend) e next lint + jest (frontend) a cada PR. Bloqueia merge de código quebrado antes de chegar à produção. Railway já faz deploy automático — o CI seria uma camada de qualidade adicional.

7.3 — Health Check configurado no Railway

Railway suporta health check endpoint nativo. Configurar para monitorar /health ou / e reiniciar automaticamente se ficar unhealthy.
