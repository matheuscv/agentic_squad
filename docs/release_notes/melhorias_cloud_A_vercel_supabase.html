Custo: $0 / mês — sem necessidade de cartão de crédito
Esta solução combina os free tiers de três plataformas. Nenhuma exige pagamento para os limites cobertos por este projeto. A única limitação relevante é a hibernação do backend no Render após 15 minutos de inatividade — a primeira requisição após hibernação demora ~20–30 segundos.
⚠️
Atenção: hibernação do Render e pausa do Supabase
O backend no Render free tier hiberna após 15 minutos sem requisição. Para uso frequente, considere upgrade para o plano Starter (~$7/mês). O Supabase pausa o projeto após 1 semana de inatividade — manter o app em uso evita isso. Ambos reiniciam automaticamente na próxima requisição.

Premissas técnicas

  • P4 — O repositório deve estar no GitHub (público ou privado — as três plataformas suportam ambos)
  • P5 — Branch principal (master) é a branch de produção. Push nela = novo deploy automático no Render (backend) e no Vercel (frontend)
  • P6 — O banco atual é SQLite local (contatos.db). Os dados de desenvolvimento não serão migrados — o banco Supabase iniciará vazio e as migrations do Alembic criarão as tabelas automaticamente
  • P7 — Será necessário criar o primeiro usuário administrador manualmente após o deploy (o sistema possui autenticação JWT sem cadastro público de admin)
  • P8 — O backend no Render free tier hiberna após 15 minutos sem requisição. A primeira requisição após hibernação demora ~20–30 segundos. Para uso pessoal/demo, isso é aceitável

Arquitetura final

Usuário no browser │ ▼ Vercel (frontend Next.js) └── URL: https://seu-app.vercel.app (sempre ativo, CDN global) │ NEXT_PUBLIC_API_URL (internet pública) ▼ Render (backend FastAPI / uvicorn) └── URL: https://seu-backend.onrender.com (hiberna após 15min) │ DATABASE_URL (internet pública → SSL) ▼ Supabase (PostgreSQL 15) └── URL: postgresql://...supabase.co:5432/postgres (sempre ativo)

Nota: nesta opção, a comunicação entre o backend (Render) e o banco (Supabase) ocorre pela internet pública com SSL, diferentemente do Railway onde os serviços se comunicam via rede privada interna. Para o volume de dados deste projeto, essa diferença não é perceptível na prática.

Etapa 1 — Preparação local do código

Garantir que o código esteja pronto para produção antes de qualquer deploy

~30–60 min Claude Code executa

1.1 — Ajuste do alembic/env.py para usar variável de ambiente

O Alembic atualmente tem a URL SQLite hardcoded no alembic.ini. Será alterado o alembic/env.py para ler DATABASE_URL do ambiente em tempo de execução. Sem esse ajuste, as migrations falham ou criam tabelas no arquivo SQLite local em vez do PostgreSQL do Supabase.

1.2 — Adicionar CORS ao app/main.py

O frontend no Vercel (domínio *.vercel.app) precisa de permissão explícita para chamar a API no Render. Será adicionado o middleware CORSMiddleware do FastAPI, lendo CORS_ORIGINS de variável de ambiente. Sem isso, o browser bloqueia todas as requisições — o app fica inacessível para o usuário final.

1.3 — Criar Procfile no diretório backend/

Render detecta o Procfile automaticamente para saber como iniciar a aplicação Python.

web: uvicorn app.main:app --host 0.0.0.0 --port $PORT

1.4 — Criar arquivo .env.example em backend/ e frontend/

Documentação de referência com todas as variáveis necessárias (sem valores reais). Servirá como guia ao configurar as variáveis no painel do Render e do Vercel.

1.5 — Verificar .gitignore

Confirmar que .env e contatos.db não estão sendo versionados. Previne exposição acidental de segredos no GitHub.

1.6 — Commit e push das alterações

Criar commit com todas as mudanças de preparação e push para master no GitHub.

Etapa 2 — Configuração do banco de dados (Supabase)

Criar e configurar o PostgreSQL de produção

~10–15 min Você executa

2.1 — Criar projeto no Supabase

  • Acessar supabase.com → "New Project"
  • Escolher nome (ex.: agentic-squad-prod)
  • Definir senha do banco (guardar em local seguro)
  • Selecionar região mais próxima: South America (São Paulo) se disponível, senão US East
  • Aguardar provisionamento (~2 minutos)

2.2 — Obter a connection string

  • No painel Supabase: Project Settings → Database → Connection string → URI
  • Copiar a URL no formato: postgresql://postgres:[SENHA]@db.xxxx.supabase.co:5432/postgres
  • Guardar essa string — será usada como DATABASE_URL no Render

2.3 — Atenção ao modo de conexão

O Supabase usa connection pooling com Supavisor. Para Alembic/migrations, usar a string com Session mode (porta 5432), pois migrations precisam de transações longas. Para a aplicação, qualquer modo funciona.

3.3 — Primeiro deploy e execução das migrations

Clicar "Deploy" no Render e aguardar o build (~3–5 minutos na primeira vez). Após build bem-sucedido, executar as migrations via Render Shell:

alembic upgrade head

Isso criará todas as tabelas no banco Supabase. O projeto possui 5 migrations: initial, deletado_em, auditoria, indices, indices sort/filter.

3.4 — Verificar a API

Acessar https://agentic-squad-backend.onrender.com/docs → Swagger UI deve aparecer. Confirmar que a API está respondendo sem erros.

3.5 — Criar o primeiro usuário administrador

Via Render Shell no serviço backend, executar script de seed ou chamada ao endpoint de criação de usuário. O banco inicia vazio — sem esse passo não é possível fazer login no sistema.

Etapa 6 — Hardening básico (opcional, pós-validação)

Melhorias de segurança e operabilidade — não bloqueante

~30–60 min Opcional

6.1 — Domínio customizado

Vercel permite adicionar domínio próprio gratuitamente (precisaria comprar o domínio ~R$40–80/ano). Render também permite domínio customizado no free tier.

6.2 — GitHub Actions para CI

Adicionar workflow que roda pytest (backend) e next lint + jest (frontend) a cada PR. Bloqueia merge de código quebrado antes de chegar à produção. Render e Vercel já fazem deploy automático — o CI seria uma camada de qualidade adicional.

6.3 — Health check no Render

Configurar o endpoint de health check no painel do Render. O Render monitora automaticamente e reinicia o serviço se ficar unhealthy. Também reduz o tempo de cold start após hibernação em alguns cenários.

6.4 — Keep-alive para evitar hibernação (opcional)

Serviços como UptimeRobot (gratuito) podem fazer ping no backend a cada 14 minutos, evitando a hibernação do Render free tier. Não é necessário se o uso esporádico for aceitável.
