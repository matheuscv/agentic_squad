Fase A — Segurança crítica

Crítico ~10h estimadas

Vulnerabilidades identificadas em auditoria que devem ser tratadas antes de qualquer deploy em produção. Sistemas de adquirência operam em ambiente CDE e estão sujeitos a PCI DSS v4.0.1.

  • A.1 — Migrar JWT de localStorage para httpOnly cookie
    Hoje o token de acesso é gravado em localStorage em frontend/src/hooks/useAuth.tsx e recuperado pelo interceptor em frontend/src/services/api.ts. Isso expõe a sessão a qualquer script malicioso injetado via XSS.
    Solução: backend emite cookie Set-Cookie: token=...; HttpOnly; Secure; SameSite=Strict; Path=/; frontend remove o storage do token e passa a depender do envio automático de cookie em cada request.
    Novo achado Esforço: ~3h
  • A.2 — Falhar startup com SECRET_KEY fraca em produção
    backend/app/config.py aceita qualquer valor de SECRET_KEY, inclusive o default dev_*. Em ENV=production, validar entropia mínima (≥ 32 bytes, prefixo diferente de dev_) e abortar o boot com mensagem clara se a validação falhar.
    Novo achado Esforço: ~1h
  • A.3 — CORS configurável via variável de ambiente
    Em backend/app/main.py a lista de origens permitidas está hardcoded para http://localhost:3000. Substituir por ALLOWED_ORIGINS (CSV) lido do .env, com fallback seguro para ambiente local apenas.
    Novo achado Esforço: ~1h
  • A.4 — Rate limit composto (IP + email) no login
    Em backend/app/routers/auth.py o limite atual de 5 req/min é por IP. Atacantes podem rotacionar IPs ou fazer enumeração de usuários. Acrescentar chave composta {ip}:{email} com limite mais restritivo (ex: 3/min) para reduzir credential stuffing e enumeration.
    Novo achado Esforço: ~2h
  • A.5 — Limitar tamanho de payload e hosts confiáveis
    Adicionar TrustedHostMiddleware com lista de hosts liberados e middleware customizado que rejeite requests com Content-Length acima de um teto (ex: 1 MB para JSON). Mitiga payload bomb e ataques de Host header injection.
    Novo achado Esforço: ~2h
  • A.6 — Política de senha forte e bloqueio progressivo
    Hoje a validação de senha é mínima. Adicionar regras de complexidade (≥ 12 chars, classes mistas) no schema Zod frontend/src/lib/schemas.ts e no Pydantic do backend, mais bloqueio temporário de conta após N falhas consecutivas (ex: 10 minutos após 5 falhas).
    Novo achado Esforço: ~2h

Fase B — Observabilidade e operação

Alto ⚙️ Parcialmente concluída (B.1 + B.2) ~12h estimadas

O backend não tem logging estruturado nem captura centralizada de exceções, o que dificulta qualquer investigação de incidente. Esta fase prepara o sistema para operação real.
Status atual: B.1 e B.2 entregues (Jira SCRUM-8, regressivo 236/236, cobertura 98%). B.3, B.4 e B.5 permanecem pendentes.

  • B.1 — Logging estruturado em JSON ✅ Feito
    Adicionar logging.getLogger(__name__) em todos os services e routers. Em produção, configurar handler com formato JSON (ex: python-json-logger) incluindo request_id, user_id, route e duration_ms.
    Entregue: backend/app/logging_config.py (ContextFilter + RedactionFilter + setup JSON em PROD / texto em DEV); backend/app/middleware/request_context.py (gera/propaga X-Request-ID, mede duration_ms); logger nomeado em todos os routers/services. Doc operacional em docs/OPERACAO_LOGS.md.
    Novo achado Esforço: ~3h Jira: SCRUM-8
  • B.2 — Tratamento centralizado de exceções ✅ Feito
    Criar backend/app/exceptions.py com hierarquia de exceções de domínio (DomainError, NotFoundError, ConflictError, etc.) e registrar um exception_handler global no FastAPI que converte cada classe em HTTP status apropriado e loga o stack trace. Elimina o except Exception: pass em main.py:36.
    Entregue: backend/app/exceptions.py (DomainError, NotFoundError 404, ConflictError 409, ValidationError 422, AuthenticationError 401, AuthorizationError 403); backend/app/exception_handlers.py com handler genérico Exception → 500 sem stack trace no body; except Exception em main.py estreitado para (AttributeError, TypeError) com log. 21 testes novos (middleware/handlers/redaction/setup), regressivo 236/236, cobertura 98%.
    Novo achado Esforço: ~3h Jira: SCRUM-8
  • B.3 — Sentry no backend e frontend
    Integrar sentry-sdk[fastapi] no backend e @sentry/nextjs no frontend. Configurar sampling e tags de release. Promovido a alto pois é pré-requisito para detectar regressões introduzidas pelas mudanças da Fase A.
    Vem do v1 (5.4) Esforço: ~2h
  • B.4 — Audit log persistente para ações sensíveis
    Criar tabela audit_log (id, ator_id, acao, recurso, recurso_id, ip, user_agent, payload_hash, timestamp) registrando: login bem-sucedido e falho, criação/exclusão de usuário, mudança de role, soft delete de contato e restore. Complementa o criado_por_id/atualizado_por_id da Fase 3.2.
    Novo achado Esforço: ~3h
  • B.5 — Healthcheck e readiness endpoints
    GET /health/live (200 sempre que o processo está vivo) e GET /health/ready (verifica conexão de banco, migration up-to-date). Pré-requisito para orquestradores (Docker Compose / K8s) na Fase E.
    Novo achado Esforço: ~1h

Fase C — Qualidade de código e arquitetura

Médio ~14h estimadas

Refatorações pontuais que reduzem custo de manutenção e abrem caminho para escala. Nenhum item é bloqueante, mas a dívida cresce a cada nova feature da Fase D.

  • C.1 — Quebrar ContatoForm.tsx (377 linhas)
    Em frontend/src/components/ContatoForm.tsx, extrair lógica de validação para um hook useContatoFormValidation() e separar campos em InputField, SelectField, TextAreaField reutilizáveis. Alvo: reduzir a componente para ~150–180 linhas.
    Novo achado Esforço: ~3h
  • C.2 — Hook useDebounce reutilizável
    Em frontend/src/app/contatos/page.tsx o debounce de busca está implementado ad-hoc com useRef. Extrair para frontend/src/hooks/useDebounce.ts (assinatura useDebounce(value, delay)) e reutilizar em filtros futuros (Fase D).
    Novo achado Esforço: ~1h
  • C.3 — Memoização estratégica em ContatoTable
    O loop de 5 skeleton rows em ContatoTable.tsx re-renderiza a cada update. Envolver SkeletonRow em React.memo e, se necessário, usar useMemo na função de ordenação.
    Novo achado Esforço: ~1h
  • C.4 — Migrar SQLAlchemy .query() → 2.0 select()
    Vários módulos em backend/app/routers/ e backend/app/services/ usam o estilo legado db.query(Model). Padronizar para a API 2.0 (select(Model)) alinhada com a versão das dependências em requirements.txt.
    Novo achado Esforço: ~2h
  • C.5 — Helper de validação de unicidade
    O padrão "buscar por email e levantar 409 se existir" está duplicado entre routers/usuarios.py e services/contato_service.py. Extrair para função utilitária em app/services/_helpers.py.
    Novo achado Esforço: ~1h
  • C.6 — Índices em colunas de busca e ordenação
    Em backend/app/models/contato.py as colunas nome, email e criado_em não têm index=True. Adicionar índices via migration Alembic; impacto especialmente alto quando D.4 (filtros) e D.7 (dashboard) forem implementados.
    Novo achado Esforço: ~1h + migração
  • C.7 — Elevar cobertura de testes do frontend
    Jest + Testing Library já estão configurados, mas a cobertura aparente é < 20%. Criar testes para: fluxo de login, validação do ContatoForm, paginação/busca da listagem, modal de "alterações não salvas". Meta: ≥ 60%.
    Novo achado Esforço: ~4h
  • C.8 — Linter de segurança no pipeline
    Adicionar bandit (Python) e eslint-plugin-security ao pre-commit hook (E.5). Falhar build se houver findings de severidade ≥ MEDIUM.
    Novo achado Esforço: ~1h

Fase D — UX e novas funcionalidades

Valor de negócio ~33h estimadas

Reorganização das Fases 2 e 4 do v1 ainda não entregues, somadas a duas novas melhorias de UX identificadas em auditoria de acessibilidade e internacionalização.

  • D.1 — Ordenação de colunas na tabela
    Backend: parâmetros sort_by e sort_order em GET /contatos/. Frontend: ícone de seta nos headers com estado ASC/DESC. Depende de C.6 (índices).
    Vem do v1 (2.1) Esforço: ~4h
  • D.2 — CRUD completo de usuários
    Hoje só existe POST /usuarios/. Adicionar GET (lista + detalhe), PUT, DELETE e PATCH /usuarios/{id}/role (apenas adm). Tela frontend em /usuarios.
    Vem do v1 (2.2) Esforço: ~7h
  • D.3 — Máscara e validação de telefone
    react-input-mask já está no package.json; aplicar máscara (99) 99999-9999 em ContatoForm e validar via Zod.
    Vem do v1 (2.3) Esforço: ~2h
  • D.4 — Filtros avançados de contatos
    Painel "Filtros" colapsável: empresa, range de criação, sem email, sem telefone. Aproveitar useDebounce (C.2) e índices (C.6).
    Vem do v1 (4.1) Esforço: ~5h
  • D.5 — Exportar contatos (CSV / Excel)
    Botão "Exportar" respeita busca/filtros atuais. Backend: GET /contatos/export?format=csv com streaming response.
    Vem do v1 (4.2) Esforço: ~4h
  • D.6 — Importar contatos via CSV
    Upload com mapeamento de colunas, preview e dry-run antes do commit. Validação linha-a-linha com relatório de erros.
    Vem do v1 (4.3) Esforço: ~6h
  • D.7 — Dashboard / métricas
    Tela /dashboard: total de contatos, criados no mês, top empresas. Backend: GET /stats com agregação SQL.
    Vem do v1 (4.4) Esforço: ~5h
  • D.8 — Recuperação de senha por e-mail
    POST /auth/forgot-password → token de uso único com TTL curto; POST /auth/reset-password valida e redefine. Integrar provedor SMTP (Resend/SendGrid).
    Vem do v1 (4.5) Esforço: ~6h
  • D.9 — Tags / categorias em contatos
    Tabela tag + tabela associativa contato_tag. Filtro multi-tag na listagem.
    Vem do v1 (4.6) Esforço: ~5h
  • D.10 — Acessibilidade: aria-label em botões/ícones
    Botões com ícone-somente em ContatoForm.tsx e campo de busca em contatos/page.tsx não têm rótulo acessível. Adicionar aria-label e role onde necessário; rodar axe-core no CI.
    Novo achado Esforço: ~1h
  • D.11 — Internacionalização (i18n)
    Todos os textos estão hardcoded em PT-BR. Adotar next-intl com namespaces por feature. Não é bloqueante hoje, mas inclui-se aqui para quando houver clientes fora do Brasil.
    Novo achado Esforço: ~3h

Fase E — Infraestrutura e operação

Operacional ~18h estimadas

Pendências da Fase 5 do v1 + adições orientadas a qualidade de pipeline. A migração para PostgreSQL (E.2) deixou de ser opcional e é pré-requisito para qualquer carga real.

  • E.1 — Docker Compose com api + frontend + db
    docker-compose.yml com três serviços. Healthcheck (B.5) usado pelo depends_on: condition: service_healthy.
    Vem do v1 (5.1) Esforço: ~3h
  • E.2 — Migrar SQLite → PostgreSQL
    Atualizar DATABASE_URL, ajustar tipos (UUID nativo, JSONB para tags em D.9), rodar suíte de testes contra a nova engine. Promovido a "alta prioridade" porque SQLite não suporta a concorrência exigida por D.4–D.7.
    Vem do v1 (5.2) Esforço: ~4h + testes
  • E.3 — CI/CD com GitHub Actions
    Pipeline: lint → testes backend (pytest) → testes frontend (jest) → build Docker → deploy. Cobertura mínima como gate (≥ 80% backend, ≥ 60% frontend após C.7).
    Vem do v1 (5.3) Esforço: ~4h
  • E.4 — Dark mode
    Tailwind dark: classes + toggle no Navbar + persistência por usuário (via API) ou localStorage como fallback.
    Vem do v1 (5.5) Esforço: ~4h
  • E.5 — Pre-commit hooks (ruff, black, prettier, eslint)
    .pre-commit-config.yaml com ruff + black no backend e prettier + eslint no frontend. Bandit (C.8) também aqui. Reduz ruído de revisão e padroniza o código produzido pelos agentes.
    Novo achado Esforço: ~2h
  • E.6 — Retry e timeout no cliente HTTP
    frontend/src/services/api.ts não tem timeout customizado nem retry. Configurar timeout: 10000 e usar axios-retry para 5xx idempotentes (apenas GET, HEAD).
    Novo achado Esforço: ~1h

Recomendação de sequência

Para colocar em produção com segurança

Fase A (toda) → Fase B.1, B.2, B.5 → Fase E.2 (PostgreSQL) → Fase E.1 (Docker) → Fase E.3 (CI/CD) → Fase B.3 (Sentry) → Fase B.4 (audit log). Esse caminho atende PCI DSS v4.0.1 e habilita operação observável antes de ligar tráfego real.

Para evoluir produto rapidamente

Fase A.1, A.2 (mínimo de segurança) → Fase C.1, C.2 (preparar terreno) → Fase D.1, D.4, D.7 (ordenação, filtros, dashboard) → Fase D.5, D.6 (export/import) → restante de C e E em paralelo.

Para reduzir dívida técnica antes de novas features

Fase C inteira → Fase B.1, B.2 → Fase E.5 (pre-commit) → Fase A. Recomendado se a squad precisa estabilizar a base antes de aceitar novas histórias.
